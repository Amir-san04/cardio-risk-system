# ДИПЛОМНАЯ РАБОТА

## Тема: "Система прогнозирования кардиологических рисков на основе машинного обучения"

---

## 1. ВВЕДЕНИЕ

### 1.1 Актуальность темы

Сердечно-сосудистые заболевания (ССЗ) остаются одной из ведущих причин смертности во всем мире. По данным Всемирной организации здравоохранения, ежегодно от ССЗ умирает около 17,9 миллионов человек, что составляет 31% всех случаев смерти в мире.

Ранняя диагностика и своевременное прогнозирование рисков позволяют значительно снизить смертность и улучшить качество жизни пациентов. Современные технологии машинного обучения открывают новые возможности для автоматизированного анализа медицинских данных и прогнозирования рисков.

### 1.2 Цель работы

Разработать веб-ориентированную систему прогнозирования кардиологических рисков на основе машинного обучения с микросервисной архитектурой, обеспечивающую:
- Хранение и обработку медицинских данных
- Прогнозирование рисков с использованием ML моделей
- Ролевое управление доступом (врачи, пациенты, администраторы)
- Поддержку стандарта DICOM для медицинских изображений

### 1.3 Задачи

1. Проектирование архитектуры микросервисной системы
2. Разработка базы данных для хранения медицинских записей
3. Реализация системы аутентификации и авторизации
4. Интеграция модуля машинного обучения
5. Реализация поддержки DICOM формата
6. Разработка пользовательского интерфейса
7. Тестирование и документирование системы

### 2.3 Микросервисы

#### 2.3.1 Core Service (Основной сервис)

**Назначение:** Центральный сервис системы, управляющий пользователями, обследованиями и файлами.

**Функциональность:**
- Регистрация и аутентификация пользователей
- Управление ролями (Admin, Doctor, Patient)
- CRUD операции для обследований
- Загрузка и хранение медицинских файлов
- Парсинг DICOM метаданных
- Интеграция с Prediction Service

**Endpoints:**
```
POST   /register               - Регистрация
POST   /token                  - Логин
GET    /me                     - Текущий пользователь
GET    /users                  - Список пользователей (Admin)
GET    /doctors                - Список врачей
GET    /patients               - Список пациентов (Doctor/Admin)

POST   /examinations           - Создать обследование (Doctor)
GET    /examinations           - Список обследований
GET    /examinations/{id}      - Детали обследования

POST   /examinations/{id}/upload-file  - Загрузить файл
GET    /examinations/{id}/files        - Список файлов

POST   /examinations/{id}/predict      - Получить прогноз (Doctor/Admin)
GET    /examinations/{id}/predictions  - История прогнозов
```

**Технологии:**
- FastAPI
- SQLAlchemy + PostgreSQL
- PyDicom (парсинг DICOM)
- MinIO Client (хранение файлов)
- JWT + bcrypt (безопасность)

#### 2.3.2 Prediction Service (ML Сервис)

**Назначение:** Сервис машинного обучения для прогнозирования кардиологических рисков.

**Функциональность:**
- Получение данных пациента
- Предобработка данных
- Применение ML модели
- Возврат прогноза с уровнем риска и рекомендациями

**Endpoints:**
```
POST   /predict         - Получить прогноз
POST   /batch-predict   - Пакетное прогнозирование
GET    /health          - Проверка состояния
```

**Входные параметры модели:**
```python
{
    "age": int,           # Возраст пациента
    "sex": int,           # Пол (0 = female, 1 = male)
    "cholesterol": int,   # Уровень холестерина
    "fbs": int,           # Сахар крови натощак (0/1)
    "restecg": int        # ЭКГ покоя (0/1/2)
}
```

**Выходные данные:**
```python
{
    "risk_level": str,       # "Low Risk" | "Medium Risk" | "High Risk"
    "risk_score": float,     # Вероятность 0.0-1.0
    "recommendations": [str] # Список рекомендаций
}
```

### 3.3 Связи между таблицами

1. **users (1) ↔ (N) examinations**
   - Один врач может проводить много обследований
   - Один пациент может иметь много обследований

2. **examinations (1) ↔ (N) medical_files**
   - К одному обследованию можно прикрепить несколько файлов

3. **examinations (1) ↔ (N) risk_predictions**
   - Для одного обследования может быть несколько прогнозов
   - (например, при повторном анализе с новой версией модели)

## 4. БЕЗОПАСНОСТЬ

### 4.1 Аутентификация

Система использует JWT (JSON Web Tokens) для аутентификации пользователей.

**Процесс аутентификации:**

1. Пользователь отправляет email и пароль на `/token`
2. Сервер проверяет пароль (bcrypt hash)
3. При успехе генерируется JWT токен
4. Токен возвращается клиенту
5. Клиент включает токен в заголовок каждого запроса:
   ```
   Authorization: Bearer <token>
   ```

**Структура JWT токена:**
```json
{
  "sub": "doctor@hospital.com",
  "role": "doctor",
  "user_id": 123,
  "exp": 1234567890
}
```

### 4.2 Авторизация (RBAC)

Система реализует ролевую модель доступа (Role-Based Access Control).

#### Роли и их права:

| Действие | Admin | Doctor | Patient |
|----------|-------|--------|---------|
| Просмотр всех пользователей | ✅ | ❌ | ❌ |
| Просмотр списка врачей | ✅ | ✅ | ✅ |
| Просмотр списка пациентов | ✅ | ✅ | ❌ |
| Создание обследования | ✅ | ✅ | ❌ |
| Просмотр своих обследований | ✅ | ✅ | ✅ |
| Загрузка файлов | ✅ | ✅ | ❌ |
| Получение прогноза | ✅ | ✅ | ❌ |
| Просмотр результатов | ✅ | ✅ | ✅* |

*Пациент видит только свои результаты

### 4.3 Хранение паролей

Пароли никогда не хранятся в открытом виде. Используется библиотека **bcrypt** с cost factor = 12.

**Процесс:**
```python
# Регистрация
password = "user_password"
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
# Сохраняется в БД: $2b$12$...

# Проверка при входе
is_valid = bcrypt.checkpw(password.encode(), stored_hash)
```

## 5. РАБОТА С DICOM

### 5.1 Стандарт DICOM

DICOM (Digital Imaging and Communications in Medicine) - международный стандарт для хранения и передачи медицинских изображений.

**Особенности формата:**
- Бинарный формат
- Содержит как изображение, так и метаданные
- Структура тегов (Tag Groups)
- Поддержка различных модальностей (CT, MRI, X-Ray и т.д.)

### 5.2 Парсинг DICOM файлов

Система использует библиотеку **pydicom** для извлечения метаданных.

**Извлекаемые данные:**

**Информация о пациенте:**
- Patient Name
- Patient ID
- Patient Birth Date
- Patient Sex
- Patient Age

**Информация об исследовании:**
- Study Date
- Study Time
- Study Description
- Modality (CT, MR, US и т.д.)

**Технические параметры:**
- Image Size (Rows × Columns)
- Pixel Spacing
- Slice Thickness
- Bits Allocated
- Manufacturer
- Equipment Model

**Параметры облучения (для CT/X-Ray):**
- KVP (напряжение трубки)
- Exposure Time
- X-Ray Tube Current

### 5.3 Хранение DICOM файлов

**Архитектура хранения:**

```
MinIO (S3 Storage)
└── medical-files (bucket)
    ├── patient_123/
    │   ├── exam_001_scan.dcm
    │   ├── exam_001_report.pdf
    │   └── exam_002_xray.dcm
    └── patient_456/
        └── exam_003_mri.dcm

PostgreSQL
└── medical_files (table)
    ├── id: 1
    ├── file_url: "s3://medical-files/patient_123/exam_001_scan.dcm"
    ├── file_type: "DICOM"
    └── file_metadata: { ... JSONB ... }
```

**Преимущества разделения:**
- База данных хранит только метаданные (быстрый поиск)
- Файлы хранятся в объектном хранилище (масштабируемость)
- Легко делать бэкапы
- Можно интегрироваться с облачными сервисами (AWS S3, Google Cloud Storage)

```

```

## 7. РАЗВЕРТЫВАНИЕ

### 7.1 Docker Compose

Система развертывается с помощью Docker Compose. Все сервисы запускаются одной командой:

```bash
docker-compose up --build
```

**Конфигурация docker-compose.yml:**
- PostgreSQL (порт 5432)
- MinIO (порты 9000, 9001)
- Core Service (порт 8000)
- Prediction Service (порт 8001)

### 7.2 Переменные окружения

```env
# Database
DATABASE_URL=postgresql://user:password@db:5432/cardio_db

# Security
SECRET_KEY=your-secret-key-here

# MinIO
MINIO_ENDPOINT=minio:9000
MINIO_ACCESS_KEY=minioadmin
MINIO_SECRET_KEY=minioadmin123
MINIO_BUCKET=medical-files
```

### 7.3 Масштабирование

Благодаря микросервисной архитектуре, система легко масштабируется:

**Горизонтальное масштабирование:**
- Можно запустить несколько экземпляров Core Service за Load Balancer
- Prediction Service можно масштабировать независимо

**Вертикальное масштабирование:**
- PostgreSQL можно перенести на более мощный сервер
- MinIO поддерживает распределенное хранилище
